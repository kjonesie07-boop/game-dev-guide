Online safety overview

Game development involves a lot of online activity — downloading software, joining communities, publishing your games, asking for help. Most of it is totally fine. Here's what to watch out for.

Protecting your personal information

🌐
The golden rule — once it's online, treat it as public forever
Anything you publish online — code on GitHub, a game on itch.io, a post on Reddit, a comment in a Discord server — should be treated as permanently public, even if the platform has privacy settings. Screenshots get taken, pages get archived, and things can spread beyond where you posted them. Before you publish, post, or upload anything anywhere, ask yourself: would I be comfortable if my aunt, my school, or a complete stranger saw this? If the answer is no, don't post it. When in doubt, check with your aunt first.
✋ Acknowledgement required
  • Use a username or alias everywhere — GitHub, itch.io, Reddit, Discord. Your real name doesn't need to be public anywhere.
  • Don't put your age, school, city, or any personal details in public profile bios or game descriptions.
  • Your itch.io page (where you'll eventually publish games) is visible to the whole internet. Keep your profile minimal: alias, what games you make, nothing else.
  • If someone in a game dev community asks where you live, how old you are, or wants to move the conversation somewhere private — don't. Block and move on.
  • Game dev communities are generally very friendly. The same internet safety rules apply everywhere, though: you don't actually know who anyone is online.

Downloading software safely

  • Only download Godot from godotengine.org directly. Search results sometimes show ads for fake lookalike sites at the very top.
  • Only download VS Code from code.visualstudio.com directly.
  • Extensions inside VS Code are safe — only install them from VS Code's built-in Extensions panel, not from random websites.
  • Never download "cracked" or "free premium" versions of any software. They almost always contain malware. Every tool in this guide is genuinely free — you don't need cracked anything.
  • If someone on Discord or Reddit DMs you with a link to "free game assets" or "a better version of Godot," don't click it. These are almost always phishing or malware.
  • When in doubt about any download, ask your aunt before clicking. It takes 30 seconds and can save a lot of headaches.

Using free game assets (art, music, sounds)

  • Free assets still have licenses. Most let you use them freely — some ask for credit. Read the license before using anything.
  • Stick to reputable free asset sites: Kenney.nl (CC0 — no restrictions at all), OpenGameArt.org (read each license), Freesound.org (sound effects).
  • Don't use assets ripped from commercial games (Mario sprites, Zelda music) in anything you publish online. That's copyright infringement.
  • Don't use random images from Google Images in your games. Most of those are copyrighted.

Accounts and passwords

  • Use a different password for each account (GitHub, itch.io, etc.). A free password manager like Bitwarden makes this easy.
  • Turn on two-factor authentication (2FA) on any account that offers it — especially GitHub. It's a free extra layer of security.
  • Your GitHub account contains all your game code. Treat its password like it matters.
  • If you ever get an email saying your account was hacked or asking you to "verify your credentials," don't click any links in it. Go to the site directly by typing the URL yourself.
🛡️
One rule that covers almost everything
If something feels off, weird, or too good to be true — it probably is. Free premium software, unsolicited DMs with helpful links, strangers weirdly interested in you personally. Trust that feeling. Close the tab, and ask your aunt if you're unsure.

What to install

Install in this order
Godot first, then VS Code + the godot-tools extension, then Python if you want it. Git last, once you have something worth saving. Everything below is free.
🎮
Godot Engine 4 Free forever
The game engine you'll build everything in. Free, open source, beginner-friendly. Download only from godotengine.org — click Download and grab the Standard version (ignore the one that says ".NET", that's for a different coding language you don't need yet). Just unzip the file and run it — no installer needed. No account required.
🛡️
Safe Godot download checklist
Before downloading, confirm the URL bar says exactly godotengine.org. Search engine ads sometimes show fake sites at the top of results — scroll past them. The file should be a .zip or .exe from the official site only.
✋ Acknowledgement required
💻
VS Code Free
Best code editor to use alongside Godot. Download only from code.visualstudio.com. After installing: open VS Code → click the Extensions icon (four squares in left sidebar) → search godot-tools → Install. This makes VS Code understand GDScript.
🐍
Python 3 Optional
Good for quick text experiments before you're deep in Godot. Download only from python.org — click the big yellow "Download Python" button for the latest version. On Windows, during the install there's a checkbox at the bottom that says "Add Python to PATH" — make sure that is checked before clicking Install, otherwise Python won't work properly.
📦
Git + GitHub account Free
Saves your game's history so you can never permanently break something. Create a free account at github.com, then download GitHub Desktop — it's a visual app that lets you save and sync your game projects with clicks instead of typing commands. Much friendlier than the command line version.
🛡️
Setting up your GitHub profile safely
When creating your GitHub account: use an alias as your username, not your real name. Don't add your school, city, or age to your profile. Your code repositories are public by default — that's normal and fine, just don't put personal info in your code comments or readme files. Also go to github.com → profile picture → Settings → Emails and make sure "Keep my email addresses private" is checked — otherwise your email can show up publicly in your commit history.

Your first conversation with Claude

The key mindset shift
Claude isn't a search engine — you're working through problems together. The more you explain what you're trying to do and what you've already tried, the better the help you get. Treat it like explaining your code to a smart friend who can write code back.

Below are the six most useful conversation types for game dev. Click each one to see an example prompt you can copy and use directly.

Give Claude the genre, mechanics, and your skill level. Ask for a plan and explanation before any code — you'll understand what you're building and catch misunderstandings early.
Prompt to use
I want to make a top-down shooter in Godot 4. The player looks at the mouse cursor and shoots bullets toward it. Enemies spawn from the edges of the screen and walk toward the player. When a bullet hits an enemy, it disappears and the enemy takes damage. I'm a complete beginner — I've watched a couple Godot tutorials but haven't finished a game yet. Can you: 1. Tell me what scenes and nodes I'll need 2. Give me the player movement and aiming code first 3. Explain what each line does as you write it
What good output looks like
C
Great starting point! Here's the scene structure you'll need:

Player scene: CharacterBody2D → Sprite2D + CollisionShape2D
Bullet scene: Area2D → Sprite2D + CollisionShape2D
Enemy scene: CharacterBody2D → Sprite2D + CollisionShape2D

Let's start with player movement. Here's the code with each line explained:
extends CharacterBody2D # How fast the player moves in pixels per second const SPEED = 200.0 func _physics_process(delta): # Read keyboard input as a direction vector (-1 to 1) var direction = Input.get_vector( "ui_left", "ui_right", "ui_up", "ui_down" ) velocity = direction * SPEED move_and_slide() # moves and handles wall collisions
Always paste the exact error text AND your full script. Say what you expected and what happened instead. The error message is almost always the most important clue.
Prompt to use
I'm getting this error in Godot when I run my game: ERROR: Invalid get index 'velocity' on base 'Nil'. at: _physics_process (res://player.gd:12) I expected the player to move when I press arrow keys, but the game crashes on startup instead. Here's my player.gd script: [paste your entire script here] What's causing this and how do I fix it?
Always copy the exact error text
From Godot's Output panel at the bottom of the editor. The red lines are errors. Select all the red text, copy it, and paste it into Claude exactly as-is. "It doesn't work" is much harder to debug than the actual error message.
Never copy-paste code you don't understand. This is how actual learning happens — not by collecting working code, but by understanding it well enough to write it yourself eventually.
Prompt to use
You just gave me this code and I have questions: [paste the code Claude gave you] Can you explain: 1. What does "delta" mean and why do we multiply velocity by it? 2. What does get_global_mouse_position() actually return? 3. Why does look_at() point the sprite the right direction automatically? Explain it like I've never programmed before.
Always describe your current setup before asking for something new. This stops Claude from giving you code that conflicts with what you already built.
Prompt to use
I want to add a double-jump to my platformer. Here's what I already have: [paste your current player.gd] Right now the player jumps once with spacebar and can't jump again until landing. I want: - First jump works as normal - Pressing space again while in the air gives one extra jump - No more jumping after the second until landing - Maybe a small visual indicator when double-jump is available Can you modify my existing script to add this?
Claude isn't just for code — use it to brainstorm mechanics, balance difficulty, and design enemies before you build them. Thinking before building saves a lot of rework.
Prompt to use
I'm designing enemies for my dungeon game. I want 3 or 4 types that feel different to fight and that a beginner could actually code in Godot 4. My player has a sword that swings in an arc in front of them. For each enemy, describe: - What it looks like (simple shapes I can make without art skills) - How it moves and attacks - What makes it interesting to fight - How hard it is to code (beginner / medium / hard) Don't write any code yet — just help me think through the design first.
For learning something entirely new — audio, saving data, adding a HUD — ask for a walkthrough rather than just finished code. You'll understand what you built and know how to fix it when it breaks.
Prompt to use
I want to add sound effects to my Godot game. I've never done audio in Godot before. Walk me through the entire process from scratch: 1. What node type do I use for sound? 2. Where does it go in the scene tree? 3. How do I import an audio file? 4. How do I play a sound when the player jumps vs when they get hit? Go step by step and tell me exactly what to click in the Godot editor. I'm using Godot 4.
Paste your scripts into the Project knowledge base
Inside your Claude Project, there's a section to add files or text. Paste your main game scripts there (player.gd, enemy.gd) so Claude always has them as context — you won't need to paste them into every single conversation.

A
Scene panel — your game's "family tree." Every object is a node. Nodes can have children inside them. Click a node to select it. Right-click to add children or delete.
B
Viewport — the visual preview of your game. Drag objects here to position them. What you see is what the player sees when playing.
C
Inspector — properties of the selected node: position, color, speed, size. Change most basic settings here without writing any code.
D
Filesystem — all your project files: scripts (.gd), scenes (.tscn), images, audio. Like File Explorer but only for your game.
📺
E — Output panel
At the bottom of the screen. When your game crashes, the error appears here in red. This is the first place to look. Copy that red text and paste it directly into Claude. White text is normal output from your code's print() calls.

The three most important concepts in Godot

1
Nodes — the building blocks of everything
Every object in Godot is a Node. CharacterBody2D is a physics body that can move. Sprite2D is an image. CollisionShape2D is an invisible hitbox. You combine nodes into a tree to build things. A "Player" is a CharacterBody2D with a Sprite2D and a CollisionShape2D as children inside it.
2
Scenes — reusable game objects
A Scene is a saved group of nodes. You make a Player scene once and drop it into any level. Make one Enemy scene, place it 20 times in your dungeon. Change the Enemy scene once — all 20 update automatically. Scenes are saved as .tscn files.
3
Scripts — code that controls a node
Scripts are .gd files you attach to nodes. Godot calls _ready() once when the scene loads, and _process(delta) sixty times per second. Everything your game does lives inside these functions and the custom ones you write.
Right-click is your friend
Right-clicking on nodes, files, and the viewport shows a menu with the most useful actions. If you don't know what something does, right-click it first.

GDScript in 2 minutes

GDScript looks almost exactly like Python. Here's everything you need to know to get started:

extends CharacterBody2D # this script belongs to a CharacterBody2D node ## Variables — storing values var speed = 200 # create a variable called "speed" var health = 3 const JUMP = -400 # const = never changes ## _ready() runs ONCE when the scene first loads func _ready(): print("Player is ready!") ## _process(delta) runs 60 TIMES PER SECOND ## delta = tiny time since last frame (~0.016 seconds) func _process(delta): if Input.is_action_pressed("ui_right"): position.x += speed * delta # move right, scaled to time ## Functions — reusable blocks of code you write func take_damage(amount): health -= amount if health <= 0: queue_free() # removes this node from the game
05

Learning path

Follow this in order. Each phase builds on the last. Don't skip to phase 3 — the basics in phase 1 are what make phase 3 fun instead of frustrating.

Phase 1 — Weeks 1–4
Get your footing
  • Python text game (number guessing, hangman)
  • Learn: variables, loops, if/else, functions
  • Open Godot — explore every panel
  • Move a colored rectangle with the keyboard
  • Understand nodes, scenes, scripts
Phase 2 — Weeks 5–10
Finish a real game
  • Build Snake or Pong from scratch
  • Add collision, scoring, game over screen
  • Learn: signals, scene instancing, tilemaps
  • Add sound effects and basic UI
  • Export a playable .exe or web build
Phase 3 — Weeks 11+
Make what you want
  • Pick a genre you love and build it
  • Add particles, lighting, screen shake
  • Publish to itch.io for others to play
  • Join a game jam (72-hour challenge)
  • Start learning shaders
🛡️
About game jams and online competitions
Game jams (like Ludum Dare, itch.io jams) are free to enter and a great challenge. Most are run through itch.io or public Discord servers. Read the rules carefully before entering. Most are very beginner-welcoming. If a jam asks for registration fees, payment info, or any personal details beyond a username — that's not normal. Check with your aunt before entering any that seem unusual.
✋ Acknowledgement required
The most important milestone
The moment everything clicks is when you finish your first complete game — even tiny. Pong or Snake, fully working, with a score and restart button. Finishing matters more than complexity. Every game developer has a "first finished game" story.

First projects to build

Build these in order. Each teaches something specific and prepares you for the next.

🎲
